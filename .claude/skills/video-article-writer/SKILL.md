# Video Article Writer — статья из видео

Полный цикл: видео → аудио → расшифровка → семантика для веб-статьи → статья 8000+ символов → анти-AI → редактура → SEO-пакет → горизонтальная обложка через LidFly.

Перед финальным контентом всегда сверяйся с `LEGAL.md`. Если тема регулируемая, спорная или медицинская, исправляй рискованные формулировки до сдачи.

## Главные правила

- Не выдумывай факты, которых нет в видео. Если нужен внешний факт, пометь его как требующий проверки или проверь через надёжный источник, когда это нужно для точности.
- Сохраняй авторский смысл видео, но не делай механическую расшифровку: статья должна читаться как самостоятельный веб-материал.
- Минимум статьи: 8000 символов с пробелами. Цель для сильной статьи: 9000-12000 символов, если тема позволяет.
- Считай и показывай оба числа: символы с пробелами и без пробелов.
- Обложку делай только через MCP LidFly `generate_ad_image` и только после подтверждения пользователя: сначала покажи промпт, дождись явного «да» (Шаг 9). Не используй `image_gen`, SVG, Canvas, локальное наложение текста или Magick для создания финальной картинки, если пользователь отдельно не попросил ручную доработку.
- Для обложки с надписями подавай весь промпт сразу в LidFly. Учитывай, что генератор может исказить кириллицу: делай короткие крупные надписи, но не переходи на ручное наложение без явного запроса.
- Перед началом работы обязательно уточни у пользователя формат публикации и стиль письма (Шаг 0). Без этих двух параметров не запускай расшифровку и не пиши статью.
- Все артефакты прогона складывай в одну папку `RESULTS/<basename>-<YYYYMMDD-HHMMSS>/`. Папку создавай один раз в Шаге 0 и используй её для всех последующих файлов: расшифровка, бриф, семантика, черновик, финальная статья, ревью, обложка. Не плоди файлы на верхнем уровне `RESULTS/`.

## Шаг 0. Уточнить формат и стиль

Прежде чем что-либо делать с видео, задай пользователю два вопроса одним сообщением и дождись ответа. Если пользователь уже указал что-то одно — спроси только недостающее.

### 0.1 Формат публикации

Спроси, куда пойдёт статья. Предложи варианты, но прими и кастомный ответ:

- **Блог** — личный/корпоративный блог, средняя глубина, можно личный голос, лёгкие отступления, мнение автора.
- **Сайт-документ / лендинг-материал** — продающий или продуктовый лонгрид, акцент на выгоде и CTA, структурированные блоки, минимум воды.
- **СМИ / медиа-лонгрид** — журналистская подача, факты, цитаты, независимый тон, без рекламного крена.
- **База знаний / документация** — обучающий формат, шаги, чеклисты, без «литературщины», нейтральный тон.
- **Хабр / технический разбор** — техническая глубина, код-блоки, ссылки на источники, ход мысли.
- **Колонка / эссе** — авторская позиция, повествовательная логика, меньше H2-сетки.

Под выбранный формат настрой:

- длину (в пределах ≥ 8000 символов, но колонка может быть 8000–10000, технический разбор 10000–14000);
- глубину H2/H3 (документация — много мелких блоков; колонка — крупные блоки);
- наличие FAQ (для блога/сайта-документа/базы знаний — обязательно; для колонки/медиа — по уместности);
- роль CTA (сайт-документ — явный CTA; медиа/колонка — без CTA или мягкий).

### 0.2 Стиль письма

Покажи пользователю список доступных стилей из папки `.styles/` и попроси выбрать один. Если нужно — покажи короткую выжимку выбранного стиля и подтверди.

```bash
ls .styles/
```

Кроме файлов из `.styles/`, предложи дополнительные универсальные опции, если они подходят теме:

- **Провокативно / остро** — резкие тезисы, прямая речь, спор с общепринятым, риторические вопросы. Без перехода на личности и оскорблений.
- **Нейтрально-экспертно** — спокойный голос, факты, минимум эмоций. Подходит для медиа и документации.
- **Дружелюбно-разговорно** — «на ты», простые формулировки, бытовые аналогии. Для блогов и обучающих материалов.
- **Сдержанно-корпоративно** — для крупных брендов: уважительный тон, без «ты», без сленга, без зачёркиваний.
- **Авторский по образцу** — пользователь даёт ссылку или абзац, по которому повторяем интонацию.

Если пользователь выбрал стиль из `.styles/`, прочитай файл целиком и используй как обязательный референс на всех шагах: бриф, структура, черновик, редактура. Маркеры, голос, запреты, списки клише и обязательные элементы из стиля переносятся в статью.

Если выбран универсальный стиль без файла, явно зафиксируй в брифе:

- голос (мы / я / нейтральный);
- обращение (вы / ты / без обращения);
- допустимый юмор и ирония;
- допустимые приёмы (зачёркивания, врезки, маркеры);
- запрещённые приёмы и слова.

### 0.3 Зафиксировать решение и создать папку прогона

После того как формат и стиль согласованы, создай отдельную папку для всего прогона:

```bash
RUN_DIR="RESULTS/<basename>-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$RUN_DIR"
```

`<basename>` — имя видео/расшифровки без расширения. Получившийся путь, например `RESULTS/my-video-20260505-201530/`, используется для всех артефактов прогона. Запомни этот путь и подставляй вместо `<RUN_DIR>` ниже.

Сохрани выбор в:

```text
<RUN_DIR>/article-config.md
```

Файл должен содержать:

- путь к исходному видео/расшифровке;
- путь к самой папке прогона;
- формат публикации и его параметры (длина, FAQ, CTA, глубина);
- выбранный стиль (имя файла из `.styles/` или название универсального стиля);
- ключевые правила стиля одним списком, чтобы было удобно сверяться на каждом шаге;
- любые дополнительные пожелания пользователя (тон, ограничения, площадка, дедлайн).

На все следующие шаги — Шаг 2 (расшифровка), Шаг 3 (бриф), Шаг 4 (семантика), Шаг 5 (структура), Шаг 6 (черновик), Шаг 7 (анти-AI), Шаг 8 (ревью), Шаг 9 (обложка) — этот файл является обязательным входом, а `<RUN_DIR>` — единственным местом, куда складываем артефакты. В ревью отдельно проверяй соответствие формату и стилю.

## Входные варианты

1. Если дан видеофайл `.mov`, `.mp4`, `.mkv`, `.webm`, `.avi` — извлеки аудио и расшифруй.
2. Если дан аудиофайл `.mp3`, `.wav`, `.m4a`, `.aac`, `.ogg` — загрузи его на расшифровку напрямую.
3. Если дана готовая расшифровка `.txt` или `.md` — пропусти извлечение аудио и распознавание, скопируй её в `<RUN_DIR>/transcription.txt` и используй как источник.
4. Если в `RESULTS/` уже есть прежняя расшифровка для этого `<basename>` (в старом плоском формате или в предыдущей папке прогона), предложи переиспользовать её вместо повторного распознавания — но всё равно скопируй в текущий `<RUN_DIR>/transcription.txt`, чтобы прогон был самодостаточен.

## Шаг 1. Извлечь аудио

Для видео используй `ffmpeg`:

```bash
ffmpeg -i "<video_path>" -vn -acodec libmp3lame -ab 96k -ar 16000 -ac 1 -y "/tmp/<basename>.mp3"
```

Если `ffmpeg` не установлен, сообщи, что нужен `ffmpeg`, и предложи `brew install ffmpeg`.

## Шаг 2. Расшифровать через LidFly

Если нужные audio tools не загружены, сначала найди их через `search_tools` по запросу `lidfly audio transcription request_upload_audio get_transcription`.

1. Запроси upload URL:

```js
request_upload_audio({
  filename: "<basename>.mp3",
  title: "<короткое название видео>",
  language: "ru",
  diarize: false
})
```

Для интервью или диалога поставь `diarize: true`.

2. Загрузи файл:

```bash
curl -s -T "/tmp/<basename>.mp3" "<upload_url>"
```

3. Получи текст:

```js
get_transcription({ id: "<transcription_id>" })
```

Если статус `processing`, проверь ещё раз через одну итерацию. Не запускай бесконечные sleep-loop.

4. Сохрани расшифровку в папку прогона:

```text
<RUN_DIR>/transcription.txt
```

## Шаг 3. Вытащить смысл видео

Перед семантикой сделай короткий контент-бриф:

- тема и обещание статьи;
- для кого материал;
- ключевая проблема читателя;
- главный вывод видео;
- 5-10 тезисов, которые точно должны попасть в статью;
- спорные или неполные места, которые нельзя превращать в уверенные факты;
- авторский тон: учебный, экспертный, разговорный, продающий, обзорный.
- формат публикации и стиль из `<RUN_DIR>/article-config.md` — повтори ключевые правила тут, чтобы бриф был самодостаточен.

Сохрани бриф в:

```text
<RUN_DIR>/article-brief.md
```

## Шаг 4. Собрать семантическое ядро для веб-статьи

Используй отдельный skill `semantic-core` в режиме `seo_site_core`.

Задача семантики здесь не рекламная кампания, а веб-статья:

- главный информационный ключ для H1 и первого абзаца;
- 5-10 поддерживающих ключей для H2/H3;
- 10-25 long-tail запросов для FAQ, примеров и подзаголовков;
- смежный спрос, который можно раскрыть одним-двумя абзацами;
- стоп-темы и нерелевантные интенты, которые нельзя тащить в статью;
- интент каждого кластера: обучающий, сравнительный, how-to, инструментальный, коммерческий.

Используй Wordstat через MCP `lidfly`, когда доступен. Если Wordstat недоступен, явно пометь семантику как оценочную.

Частотности и оценки спроса из семантики — материал для выбора структуры, угла и заголовков, **а не контент статьи**. Пересказывать их в тексте нельзя (см. шаг 6).

Сохрани результат:

```text
<RUN_DIR>/semantic-core.md
```

## Шаг 5. Построить структуру статьи

Перед написанием собери план, который покрывает и видео, и спрос. Структуру подгоняй под формат и стиль из `<RUN_DIR>/article-config.md`: для блога/сайта-документа/базы знаний используй полный H2-каркас и FAQ; для колонки/эссе — крупные блоки и мягкий FAQ или его отсутствие; для технического разбора — добавляй блоки с кодом, источниками и оговорками неопределённости.

Сохрани план в:

```text
<RUN_DIR>/article-outline.md
```

Базовый каркас, от которого отклоняемся под формат:

- `H1` с главным ключом и понятной выгодой;
- лид 300-600 символов, главный ключ в первых 100 словах;
- 6-10 H2, каждый закрывает отдельный интент;
- H3 только там, где нужна детализация;
- блок "Что важно запомнить" или "Коротко";
- FAQ 4-7 вопросов из long-tail семантики (если формат его требует);
- SEO title до 70 символов;
- meta description 140-160 символов;
- slug латиницей;
- 3-7 тегов или категорий.

Проверь план на каннибализацию: один H2 = один смысловой блок, без дублей. Заголовки и порядок разделов должны соответствовать выбранному стилю — например, в стиле `awaik.md` гипотезные разделы и врезки «Интересно.» обязательны, в нейтрально-экспертном — нет.

## Шаг 6. Написать черновик

Пиши как редактор, а не как стенографист. Перед началом перечитай `<RUN_DIR>/article-config.md` и держи открытым выбранный файл стиля из `.styles/` — голос, маркеры, запреты и обязательные элементы стиля имеют приоритет над общими правилами этого скилла, кроме фактологии и LEGAL.md.

Обязательные требования:

- минимум 8000 символов с пробелами;
- раскрыть тему без воды и механических повторов ключей;
- использовать тезисы видео как основу экспертности;
- добавить примеры, сценарии применения, типичные ошибки, сравнения, мини-чеклисты;
- встроить ключи естественно, без переспама;
- не пересказывать аналитику спроса: в тексте нет частотностей, слов «спрос»/«запрос» и отсылок к тому, что и как часто ищут; спрос определяет только выбор темы, угла и структуры, а инсайт о человеке формулируй без следов SEO-аналитики;
- не начинать текст клише вроде "в современном мире", "важно отметить", "следует подчеркнуть";
- не обещать гарантированный результат;
- если статья обучающая, дать читателю последовательность действий;
- если в видео есть демонстрация интерфейса, превратить её в понятные шаги;
- не использовать Markdown-таблицы без необходимости: таблица должна реально помогать сравнению;
- соблюдать выбранный стиль и формат: голос (мы/я/нейтральный), обращение (вы/ты), допустимый юмор, обязательные маркеры стиля, длину под формат, наличие/отсутствие FAQ и CTA.

Сохрани черновик:

```text
<RUN_DIR>/article-draft.md
```

## Шаг 7. Убрать AI-маркеры

Применяй skill `ai-markers-remove`, но держи публикационное качество:

- убери гладкие AI-переходы, симметрию, однотипные абзацы;
- добавь живой синтаксис и авторские повороты;
- не ломай ключевые фразы, факты, заголовки, названия брендов и CTA;
- не оставляй грубые орфографические ошибки в финале;
- если skill внёс слишком заметные опечатки, исправь их на редакторском шаге.

Сохрани финальную статью:

```text
<RUN_DIR>/article.md
```

## Шаг 8. Редакторское ревью

Сделай ревью как редактор перед сдачей и внеси применимые правки.

Проверь:

- статья отвечает интенту главного ключа;
- первые 2 абзаца объясняют ценность и не тянут время;
- структура логична без повторов;
- каждый H2 раскрыт достаточно глубоко;
- тезисы видео не искажены;
- нет неподтверждённых обещаний, юридических рисков и запрещённых формулировок из `LEGAL.md`;
- ключи распределены естественно;
- текст звучит как человек, но без неряшливости;
- есть конкретика: шаги, примеры, ошибки, выводы;
- FAQ не повторяет основной текст дословно;
- SEO title/meta не кликбейтные и не длиннее лимитов;
- статья соответствует формату из `<RUN_DIR>/article-config.md`: длина, наличие FAQ/CTA, глубина H2/H3;
- стиль выдержан: голос, обращение, маркеры, обязательные элементы и запреты выбранного стиля. Если стиль `awaik.md` — проверь наличие маркеров «Итак,», «Поехали.», «Интересно.», «Вывод.», «ИТОГ.», конкретных чисел, признания ограничений и приглашения в обсуждение. Если выбран другой стиль — сверь по его требованиям.

Сохрани отчёт:

```text
<RUN_DIR>/editor-review.md
```

Если правки были существенные, обнови `<RUN_DIR>/article.md` после ревью.

## Шаг 9. Сгенерировать горизонтальную картинку через LidFly

Обложку генерируем только через MCP `lidfly` и только после подтверждения пользователя. Сначала собери промпт, покажи его, дождись «да» — и лишь потом вызывай `generate_ad_image`.

### 9.1 Собрать промпт

Промпт должен включать:

- формат: горизонтальная обложка 16:9, 1920×1080;
- стиль под тему статьи и бренд проекта;
- главный визуальный объект;
- фон, свет, палитру;
- 1-3 короткие надписи на картинке, например H1/главный оффер/лейбл урока;
- требование: крупная читабельная типографика, без мелкого текста;
- отсутствие лишних логотипов, водяных знаков и случайных букв.

Не генерируй картинку сам и не накладывай текст локально. Надписи должны быть частью промпта к LidFly.

### 9.2 Показать промпт и спросить подтверждение

Покажи пользователю готовый промпт и формат, дождись ответа:

> Промпт для обложки:
> ```
> <промпт>
> ```
>
> Формат: `lidfly_hero_1920x1080` (16:9, 1920×1080). Генерировать? (да / нет / править промпт)

- «нет» → пропусти генерацию, в финальной выдаче (Шаг 10) отметь, что обложка не делалась;
- «править» → переработай промпт и снова покажи;
- «да» → переходи к 9.3.

Без явного «да» `generate_ad_image` не вызывай.

### 9.3 Сгенерировать (после подтверждения)

```js
generate_ad_image({
  format: "lidfly_hero_1920x1080",
  prompt: "<согласованный промпт>"
})
```

Перед генерацией можно вызвать `get_recent_images({ limit: 10 })`, чтобы не плодить дубликаты. Если `generate_ad_image` вернул 504, не ретраить сразу: сначала проверить `get_recent_images`.

Сохрани URL и, если нужно, локальную копию:

```text
<RUN_DIR>/cover-url.txt
<RUN_DIR>/cover.png
```

## Шаг 10. Финальная выдача

В конце отдай кратко:

- путь к папке прогона `<RUN_DIR>` — там лежит всё;
- путь к финальной статье `<RUN_DIR>/article.md`;
- путь к расшифровке `<RUN_DIR>/transcription.txt`;
- количество символов с пробелами и без;
- главный ключ и 5-10 поддерживающих;
- SEO title;
- meta description;
- путь к редакторскому ревью `<RUN_DIR>/editor-review.md`;
- URL обложки LidFly и локальный путь, если скачивал;
- что осталось проверить вручную, если есть риски.

Ожидаемая структура папки прогона по итогу:

```text
<RUN_DIR>/
├── article-config.md     # формат + стиль (Шаг 0)
├── transcription.txt     # расшифровка видео (Шаг 2)
├── article-brief.md      # контент-бриф (Шаг 3)
├── semantic-core.md      # семантика (Шаг 4)
├── article-outline.md    # план статьи (Шаг 5)
├── article-draft.md      # черновик до анти-AI (Шаг 6)
├── article.md            # финальная статья (Шаг 7, обновляется в Шаге 8)
├── editor-review.md      # ревью (Шаг 8)
├── cover-url.txt         # URL обложки (Шаг 9)
└── cover.png             # локальная копия обложки, если скачана
```

Не вставляй всю статью в чат, если она длинная. Достаточно структуры, файлов и короткого summary.

## Критерии топ-качества

- Статья не пересказывает видео подряд, а превращает его в полезный веб-материал.
- В первом экране понятны тема, ценность и кому это нужно.
- Есть собственная редакторская логика: что главное, что второстепенно, какие ошибки читателя предупредить.
- Семантика влияет на структуру, а не просто добавлена списком в конец.
- Аналитика спроса не пересказана в тексте: частотности и популярность запросов превращены в угол и инсайт, а не в справку «сколько раз ищут».
- Текст можно публиковать без стыда: нет AI-глянца, но и нет намеренной неряшливости.
- Обложка соответствует теме статьи и содержит короткий читаемый месседж.
