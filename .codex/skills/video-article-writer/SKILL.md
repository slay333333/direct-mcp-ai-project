# Video Article Writer — статья из видео

Полный цикл: видео → аудио → расшифровка → выбор ресурса и стиля → семантика для веб-статьи → статья 8000+ символов → анти-AI → редактура → SEO-пакет → горизонтальная обложка через Codex imagegen.

Перед финальным контентом всегда сверяйся с `LEGAL.md`. Если тема регулируемая, спорная или медицинская, исправляй рискованные формулировки до сдачи.

## Главные правила

- Не выдумывай факты, которых нет в видео. Если нужен внешний факт, пометь его как требующий проверки или проверь через надёжный источник, когда это нужно для точности.
- Сохраняй авторский смысл видео, но не делай механическую расшифровку: статья должна читаться как самостоятельный веб-материал.
- Минимум статьи: 8000 символов с пробелами. Цель для сильной статьи: 9000-12000 символов, если тема позволяет.
- Считай и показывай оба числа: символы с пробелами и без пробелов.
- Обложку делай через Codex: используй skill `imagegen` и встроенный tool `image_gen`. Не используй LidFly/VK image API (`generate_ad_image`, `get_recent_images` и аналоги) для создания картинки.
- Перед каждым созданием или повторной генерацией обложки покажи пользователю финальный промпт и дождись явного подтверждения. Если подтверждения нет — остановись на промпте.
- Если нужны надписи, согласуй точный текст заранее. Для кириллицы предпочтительно сгенерировать фон/визуал без текста и наложить текст локально только после отдельного согласования макета.
- Перед началом работы обязательно уточни ресурс публикации, стиль из `.styles/` и режим подачи (Шаг 0). Без этих параметров не запускай расшифровку и не пиши статью.
- Все артефакты прогона складывай в одну папку `RESULTS/<basename>-<YYYYMMDD-HHMMSS>/`. Папку создавай один раз в Шаге 0 и используй её для всех последующих файлов: расшифровка, бриф, семантика, структура, черновик, финальная статья, ревью, обложка. Не плоди файлы на верхнем уровне `RESULTS/`.

## Шаг 0. Спросить ресурс, стиль и режим подачи + создать папку прогона

До структуры и черновика обязательно выясни редакционные параметры статьи. Если пользователь уже указал что-то — не переспрашивай, фиксируй в конфиге.

### 0.1 Прочитать список доступных стилей

```bash
find .styles -maxdepth 1 -type f -name '*.md' -print
```

Покажи пользователю варианты по имени файла без `.md`. Если выбранный стиль найден, прочитай соответствующий файл из `.styles/` и применяй его как обязательный style guide на всех шагах: бриф, структура, черновик, редактура, SEO-поля и промпт обложки. Если пользователь выбрал кастомный стиль не из `.styles`, используй его как текстовый style brief и явно отметь, что файла стиля нет.

### 0.2 Спросить параметры одним сообщением

- для какого ресурса пишем статью: домен, блог, медиа, раздел сайта, канал или клиентский проект;
- какой стиль использовать из `.styles/` или кастомный;
- какой режим подачи выбрать.

Режимы подачи:

- `провокативно` — сильный конфликт, острые формулировки, цепкий заход; без кликбейта, манипуляций и юридически опасных обещаний;
- `технически` — термины, механика, архитектура, цифры, аккуратные оговорки, минимум эмоций;
- `научпоп` — сложное простым языком, аналогии, объяснение механизма, живые примеры;
- `практично` — how-to, шаги, чеклисты, типовые ошибки, конкретные действия;
- `экспертно-редакционно` — спокойная авторская колонка, аргументы, выводы, меньше рекламного тона;
- `кейс/аналитика` — ситуация, данные, причины, последствия, выводы для похожих случаев;
- `бренд-медиа` — нативная статья под ресурс, полезность впереди продажи, мягкая связь с продуктом.

Если пользователь просит несколько вариантов, подготовь 2-3 коротких захода или плана в разных режимах и попроси выбрать один перед полным черновиком.

### 0.3 Создать папку прогона и зафиксировать конфиг

После того как ресурс, стиль и режим согласованы, создай отдельную папку для всего прогона:

```bash
RUN_DIR="RESULTS/<basename>-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$RUN_DIR"
```

`<basename>` — имя видео/расшифровки без расширения. Получившийся путь, например `RESULTS/my-video-20260505-201530/`, используется для всех артефактов прогона. Запомни этот путь и подставляй вместо `<RUN_DIR>` ниже.

Сохрани конфиг в:

```text
<RUN_DIR>/article-config.md
```

Файл должен содержать:

- путь к исходному видео/расшифровке;
- путь к самой папке прогона;
- ресурс публикации (домен, блог, медиа, канал) и его аудитория;
- выбранный стиль (имя файла из `.styles/` или название кастомного стиля) + ключевые правила стиля одним списком;
- выбранный режим подачи и как он влияет на тон;
- параметры формата: длина, наличие FAQ, роль CTA, глубина H2/H3;
- любые дополнительные пожелания пользователя (тон, ограничения, площадка, дедлайн).

На все следующие шаги — Шаг 2 (расшифровка), Шаг 3 (бриф), Шаг 4 (семантика), Шаг 5 (структура), Шаг 6 (черновик), Шаг 7 (анти-AI), Шаг 8 (ревью), Шаг 9 (обложка) — этот файл является обязательным входом, а `<RUN_DIR>` — единственным местом, куда складываем артефакты. В ревью отдельно проверяй соответствие ресурсу, стилю и режиму.

## Входные варианты

1. Если дан видеофайл `.mov`, `.mp4`, `.mkv`, `.webm`, `.avi` — извлеки аудио и расшифруй.
2. Если дан аудиофайл `.mp3`, `.wav`, `.m4a`, `.aac`, `.ogg` — загрузи его на расшифровку напрямую.
3. Если дана готовая расшифровка `.txt` или `.md` — пропусти извлечение аудио и распознавание, скопируй её в `<RUN_DIR>/transcription.txt` и используй как источник.
4. Если в `RESULTS/` уже есть прежняя расшифровка для этого `<basename>` (в старом плоском формате или в предыдущей папке прогона), предложи переиспользовать её вместо повторного распознавания — но всё равно скопируй в текущий `<RUN_DIR>/transcription.txt`, чтобы прогон был самодостаточен.

## Шаг 1. Извлечь аудио

Для видео используй `ffmpeg`:

```bash
ffmpeg -i "<video_path>" -vn -acodec libmp3lame -ab 96k -ar 16000 -ac 1 -y "/tmp/<basename>.mp3"
```

Если `ffmpeg` не установлен, сообщи, что нужен `ffmpeg`, и предложи `brew install ffmpeg`.

## Шаг 2. Расшифровать через LidFly

Если нужные audio tools не загружены, сначала найди их через `search_tools` по запросу `lidfly audio transcription request_upload_audio get_transcription`.

1. Запроси upload URL:

```js
request_upload_audio({
  filename: "<basename>.mp3",
  title: "<короткое название видео>",
  language: "ru",
  diarize: false
})
```

Для интервью или диалога поставь `diarize: true`.

2. Загрузи файл:

```bash
curl -s -T "/tmp/<basename>.mp3" "<upload_url>"
```

3. Получи текст:

```js
get_transcription({ id: "<transcription_id>" })
```

Если статус `processing`, проверь ещё раз через одну итерацию. Не запускай бесконечные sleep-loop.

4. Сохрани расшифровку в папку прогона:

```text
<RUN_DIR>/transcription.txt
```

## Шаг 3. Вытащить смысл видео

Перед семантикой сделай короткий контент-бриф:

- ресурс публикации и его аудитория;
- выбранный стиль из `.styles` или кастомный style brief;
- выбранный режим подачи и как он влияет на тон статьи;
- тема и обещание статьи;
- для кого материал;
- ключевая проблема читателя;
- главный вывод видео;
- 5-10 тезисов, которые точно должны попасть в статью;
- спорные или неполные места, которые нельзя превращать в уверенные факты;
- авторский тон с учётом ресурса: учебный, экспертный, разговорный, продающий, обзорный.
- ключевые правила формата и стиля из `<RUN_DIR>/article-config.md` — повтори их тут, чтобы бриф был самодостаточен.

Сохрани бриф в:

```text
<RUN_DIR>/article-brief.md
```

## Шаг 4. Собрать семантическое ядро для веб-статьи

Используй отдельный skill `semantic-core` в режиме `seo_site_core`.

Задача семантики здесь не рекламная кампания, а веб-статья:

- главный информационный ключ для H1 и первого абзаца;
- 5-10 поддерживающих ключей для H2/H3;
- 10-25 long-tail запросов для FAQ, примеров и подзаголовков;
- смежный спрос, который можно раскрыть одним-двумя абзацами;
- стоп-темы и нерелевантные интенты, которые нельзя тащить в статью;
- интент каждого кластера: обучающий, сравнительный, how-to, инструментальный, коммерческий.

Используй Wordstat через MCP `lidfly`, когда доступен. Если Wordstat недоступен, явно пометь семантику как оценочную.

Сохрани результат:

```text
<RUN_DIR>/semantic-core.md
```

## Шаг 5. Построить структуру статьи

Перед написанием собери план, который покрывает и видео, и спрос. Структуру подгоняй под ресурс, стиль и режим подачи из `<RUN_DIR>/article-config.md`: для блога/сайта-документа/базы знаний используй полный H2-каркас и FAQ; для колонки/эссе — крупные блоки и мягкий FAQ или его отсутствие; для технического разбора — добавляй блоки с кодом, источниками и оговорками неопределённости.

Базовый каркас, от которого отклоняемся под формат:

- ресурс публикации, выбранный стиль и режим подачи;
- `H1` с главным ключом и понятной выгодой;
- лид 300-600 символов, главный ключ в первых 100 словах;
- 6-10 H2, каждый закрывает отдельный интент;
- H3 только там, где нужна детализация;
- блок "Что важно запомнить" или "Коротко";
- FAQ 4-7 вопросов из long-tail семантики (если формат его требует);
- SEO title до 70 символов;
- meta description 140-160 символов;
- slug латиницей;
- 3-7 тегов или категорий.

Проверь план на каннибализацию: один H2 = один смысловой блок, без дублей. Заголовки и порядок разделов должны соответствовать выбранному стилю.

Сохрани план в:

```text
<RUN_DIR>/article-outline.md
```

## Шаг 6. Написать черновик

Пиши как редактор, а не как стенографист. Перед началом перечитай `<RUN_DIR>/article-config.md` и держи открытым выбранный файл стиля из `.styles/` — голос, маркеры, запреты и обязательные элементы стиля имеют приоритет над общими правилами этого скилла, кроме фактологии и LEGAL.md.

Обязательные требования:

- следовать выбранному style guide из `.styles` и режиму подачи;
- адаптировать лексику, глубину объяснений и CTA под ресурс публикации;
- минимум 8000 символов с пробелами;
- раскрыть тему без воды и механических повторов ключей;
- использовать тезисы видео как основу экспертности;
- добавить примеры, сценарии применения, типичные ошибки, сравнения, мини-чеклисты;
- встроить ключи естественно, без переспама;
- не начинать текст клише вроде "в современном мире", "важно отметить", "следует подчеркнуть";
- не обещать гарантированный результат;
- если статья обучающая, дать читателю последовательность действий;
- если в видео есть демонстрация интерфейса, превратить её в понятные шаги;
- не использовать Markdown-таблицы без необходимости: таблица должна реально помогать сравнению.

Сохрани черновик:

```text
<RUN_DIR>/article-draft.md
```

## Шаг 7. Убрать AI-маркеры

Применяй skill `ai-markers-remove`, но держи публикационное качество:

- убери гладкие AI-переходы, симметрию, однотипные абзацы;
- добавь живой синтаксис и авторские повороты;
- не ломай ключевые фразы, факты, заголовки, названия брендов и CTA;
- не оставляй грубые орфографические ошибки в финале;
- если skill внёс слишком заметные опечатки, исправь их на редакторском шаге.

Сохрани финальную статью:

```text
<RUN_DIR>/article.md
```

## Шаг 8. Редакторское ревью

Сделай ревью как редактор перед сдачей и внеси применимые правки.

Проверь:

- статья отвечает интенту главного ключа;
- статья подходит выбранному ресурсу, style guide и режиму подачи;
- первые 2 абзаца объясняют ценность и не тянут время;
- структура логична без повторов;
- каждый H2 раскрыт достаточно глубоко;
- тезисы видео не искажены;
- нет неподтверждённых обещаний, юридических рисков и запрещённых формулировок из `LEGAL.md`;
- ключи распределены естественно;
- текст звучит как человек, но без неряшливости;
- есть конкретика: шаги, примеры, ошибки, выводы;
- FAQ не повторяет основной текст дословно;
- SEO title/meta не кликбейтные и не длиннее лимитов;
- статья соответствует параметрам формата из `<RUN_DIR>/article-config.md`: длина, наличие FAQ/CTA, глубина H2/H3;
- стиль и режим подачи выдержаны: голос, обращение, маркеры, обязательные элементы и запреты выбранного стиля.

Сохрани отчёт:

```text
<RUN_DIR>/editor-review.md
```

Если правки были существенные, обнови `<RUN_DIR>/article.md` после ревью.

## Шаг 9. Сгенерировать горизонтальную картинку через Codex

Используй skill `imagegen` и встроенный tool `image_gen`. Не вызывай `generate_ad_image`, `get_recent_images` или другие LidFly/VK image API для создания обложки.

Сначала подготовь промпт и покажи его пользователю:

```text
Промпт для обложки:
<полный промпт>

Формат: 16:9, 1920×1080.
Генерировать через Codex? (да / нет / править промпт)
```

Дальше действуй только по ответу:

- `нет` — пропусти обложку и зафиксируй это в финальной выдаче;
- `править промпт` — перепиши промпт и снова спроси подтверждение;
- `да` — вызови `image_gen`.

Промпт должен включать:

- формат: горизонтальная обложка 16:9, 1920×1080;
- стиль под тему статьи, выбранный ресурс, style guide и бренд проекта;
- главный визуальный объект;
- фон, свет, палитру;
- запрет на водяные знаки, лишние логотипы, случайные буквы и мелкий нечитаемый текст;
- если пользователь подтвердил надписи: точный текст 1-3 коротких фраз и требование крупной читабельной типографики.

После генерации проверь результат: тема, композиция, артефакты, случайные буквы, логотипы и качество текста, если он есть. Если результат не годится, предложи исправленный промпт, но не генерируй повторно без нового подтверждения пользователя.

Сохрани выбранный результат в папку прогона:

```text
<RUN_DIR>/cover.png
```

Если публикации нужен URL, загрузи уже созданный локальный файл через актуальный инструмент загрузки сайта. Это допустимо только как хостинг готового файла, не как API-генерация картинки.

## Шаг 10. Финальная выдача

В конце отдай кратко:

- путь к папке прогона `<RUN_DIR>` — там лежит всё;
- путь к финальной статье `<RUN_DIR>/article.md`;
- путь к расшифровке `<RUN_DIR>/transcription.txt`;
- количество символов с пробелами и без;
- ресурс публикации, стиль и режим подачи;
- главный ключ и 5-10 поддерживающих;
- SEO title;
- meta description;
- путь к редакторскому ревью `<RUN_DIR>/editor-review.md`;
- локальный путь к обложке и URL, если загружал её для публикации;
- что осталось проверить вручную, если есть риски.

Ожидаемая структура папки прогона по итогу:

```text
<RUN_DIR>/
├── article-config.md     # ресурс + стиль + режим (Шаг 0)
├── transcription.txt     # расшифровка видео (Шаг 2)
├── article-brief.md      # контент-бриф (Шаг 3)
├── semantic-core.md      # семантика (Шаг 4)
├── article-outline.md    # план статьи (Шаг 5)
├── article-draft.md      # черновик до анти-AI (Шаг 6)
├── article.md            # финальная статья (Шаг 7, обновляется в Шаге 8)
├── editor-review.md      # ревью (Шаг 8)
├── cover-url.txt         # URL обложки, если загружали для публикации
└── cover.png             # обложка, созданная Codex (Шаг 9)
```

Не вставляй всю статью в чат, если она длинная. Достаточно структуры, файлов и короткого summary.

## Критерии топ-качества

- Статья не пересказывает видео подряд, а превращает его в полезный веб-материал.
- В первом экране понятны тема, ценность и кому это нужно.
- Есть собственная редакторская логика: что главное, что второстепенно, какие ошибки читателя предупредить.
- Семантика влияет на структуру, а не просто добавлена списком в конец.
- Текст можно публиковать без стыда: нет AI-глянца, но и нет намеренной неряшливости.
- Обложка соответствует теме статьи и содержит короткий читаемый месседж.
